  <action>
In `L3ContainerPool`:
- Add instance attributes for aggregate tracking:
  - `self.completed_count: int = 0` — incremented each time a task completes (any terminal state)
  - `self.queued_count: int = 0` — tracks tasks currently waiting for a semaphore slot
  - `self._saturated: bool = False` — tracks whether pool is currently in saturation state
- In `spawn_and_monitor()`:
  1. Before `async with self.semaphore`: increment `self.queued_count`. Check if semaphore would block by attempting non-blocking acquire (`self.semaphore._value == 0` — check asyncio.Semaphore internal `_value` to detect saturation without side effects). If saturated (value is 0) and not already `self._saturated`:
     - Set `self._saturated = True`
     - Log saturation onset: `logger.warning("Pool saturation onset", extra={"project_id": self.project_id, "queued_task_id": task_id, "queue_depth": self.queued_count, "active_task_ids": self.list_active()})`
  2. Inside `async with self.semaphore` (after acquiring): decrement `self.queued_count`. If `self._saturated` and `self.semaphore._value > 0` (slot freed up beyond this one — actually: saturation is resolved when we acquired after being saturated). Better approach: if `self._saturated` was True when we entered the `async with` and now we have a slot, log resolution:
     - Set `self._saturated = False`
     - `logger.info("Pool saturation resolved", extra={"project_id": self.project_id, "task_id": task_id, "queue_depth": self.queued_count})`
  3. After task completes (after result returned, still inside semaphore context): `self.completed_count += 1`

Actually, cleaner approach to saturation detection:
  1. Before `async with self.semaphore`:
     - `self.queued_count += 1`
     - `was_saturated = self.semaphore._value == 0`  (all slots occupied)
     - If `was_saturated` and not `self._saturated`: log onset, set `self._saturated = True`
  2. After acquiring (first line inside `async with`):
     - `self.queued_count -= 1`
     - If `self._saturated`: log resolution, set `self._saturated = False`
  3. After final result: `self.completed_count += 1`

- Add `def get_pool_stats(self) -> Dict[str, Any]` method returning:
  ```python
  {
      "project_id": self.project_id,
      "active": self.get_active_count(),
      "queued": self.queued_count,
      "completed": self.completed_count,
      "max_concurrent": self.max_concurrent,
      "saturation_pct": round((self.get_active_count() / self.max_concurrent) * 100, 1),
      "saturated": self._saturated,
  }
  ```

In `PoolRegistry`:
- Add `def get_all_stats(self) -> Dict[str, Dict[str, Any]]` method that returns `{pid: pool.get_pool_stats() for pid, pool in self._pools.items()}`.

Note: These in-memory counters are for live querying by monitor. They do NOT persist to state file — per CONTEXT.md discretion, saturation events remain log-only (not persisted in state). The monitor CLI computes utilization from workspace-state.json task statuses (computed on-the-fly) since pool.py's in-memory state is only available while the pool process is running.

Add a new `pool` subcommand to `monitor.py` that displays pool utilization per project. Since pool.py's in-memory counters are only available during pool process lifetime, the monitor computes utilization on-the-fly from workspace-state.json task entries.

Add `show_pool_utilization(project_filter: Optional[str] = None) -> None` function:
1. Discover projects via `_discover_projects(project_filter)`
2. For each project, read state via `JarvisState(state_file).read_state()`
3. Compute per-project stats from task entries:
   - `active`: count of tasks with status in `{'in_progress', 'starting', 'testing'}`
   - `queued`: count of tasks with status `'pending'`
   - `completed`: count of tasks with status `'completed'`
   - `failed`: count of tasks with status in `{'failed', 'timeout'}`
   - `max_concurrent`: hardcoded 3 (Phase 23 will make this configurable per project)
   - `saturation_pct`: `round((active / max_concurrent) * 100, 1)`
4. Print colored table:
   - Header: `PROJECT  ACTIVE  QUEUED  COMPLETED  FAILED  SATURATION`
   - Per row: color-code saturation: green (0-33%), yellow (34-66%), red (67-100%)
   - If `project_filter` is None (show all): show a summary line at bottom with totals
5. If no tasks found for any project, print "No tasks found" in yellow

Per CONTEXT.md decision: default (no argument) shows all projects; passing a project ID filters. Color-code saturation levels (green/yellow/red) matching existing monitor.py color patterns.
