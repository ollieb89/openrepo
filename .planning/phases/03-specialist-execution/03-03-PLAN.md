  <name>Task 1: Implement Semantic Snapshot System (Git Staging Branch Workflow)</name>
  <files>
    orchestration/snapshot.py
  </files>
  <action>
    Create `orchestration/snapshot.py` implementing the semantic snapshot system based on Pattern 3 from 03-RESEARCH.md.

    **Functions to implement:**

    `create_staging_branch(task_id, workspace_path)`:
    - Branch name: `l3/task-{task_id}` (locked decision: staging branch model)
    - Run `git -C {workspace_path} checkout -b {branch_name} main`
    - Handle case where branch already exists (checkout existing)
    - Return branch name
    - Raise clear error if workspace is not a git repo

    `capture_semantic_snapshot(task_id, workspace_path)`:
    - Generate git diff against main: `git -C {workspace_path} diff main...HEAD`
    - Save diff to `{workspace_path}/.openclaw/snapshots/{task_id}.diff`
    - Create snapshots directory if it doesn't exist
    - Include metadata header in the diff file: task_id, branch name, timestamp, file count
    - Return snapshot path and summary (files changed count, insertions, deletions)
    - Per discretion: snapshot is captured on task completion (not every commit)

    `l2_review_diff(task_id, workspace_path)`:
    - Generate human-readable diff summary: `git -C {workspace_path} diff --stat main...l3/task-{task_id}`
    - Return the full diff and stat summary for L2 to inspect
    - Do NOT auto-merge — L2 makes the decision

    `l2_merge_staging(task_id, workspace_path)`:
    - Checkout main: `git -C {workspace_path} checkout main`
    - Merge with no-ff: `git -C {workspace_path} merge --no-ff l3/task-{task_id} -m "Merge L3 task {task_id} into main"`
    - Delete staging branch: `git -C {workspace_path} branch -d l3/task-{task_id}`
    - Return merge result (success/conflict)
    - On conflict: abort merge, return conflict details, leave branch intact for manual resolution

    `l2_reject_staging(task_id, workspace_path)`:
    - Delete the staging branch without merging
    - Update state.json task status to "rejected"
    - Return confirmation

    `cleanup_old_snapshots(workspace_path, max_snapshots=100)`:
    - Per discretion: keep last 100 snapshots, delete oldest when exceeded
    - Sort by file modification time, remove oldest

    All git operations use `subprocess.run()` with `check=True` and `capture_output=True`. Handle subprocess.CalledProcessError with meaningful error messages.

    Important: The staging branch is created by the L3 container's entrypoint.sh (Plan 02). This module provides the L2-side operations: review, merge, reject, and snapshot capture. The create_staging_branch function exists for testing and edge cases.
  </action>

  <name>Task 2: Implement CLI Monitoring Tool for Human Operator</name>
  <files>

  <action>
    Create `orchestration/monitor.py` — a CLI tool that gives the human operator real-time visibility into L3 activity. This is the Phase 3 substitute for the Phase 4 dashboard (per locked decision: CLI/log output for human operator).

    **Functions to implement:**

    `tail_state(state_file_path, interval=1.0)`:
    - Continuously poll workspace-state.json at 1-second intervals (per discretion: 1s polling)
    - On each poll, read state via JarvisState.read_state()
    - Detect changes since last poll (compare task statuses and activity_log lengths)
    - Print new activity log entries in a formatted, readable way:
      ```
      [2026-02-18 14:32:05] [task-001] [in_progress] Started code generation for UserAuth
      [2026-02-18 14:32:15] [task-001] [in_progress] Created 3 files: UserAuth.tsx, useAuth.ts, auth.test.tsx
      [2026-02-18 14:32:30] [task-002] [testing] Running test suite...
      ```
    - Show task status transitions (pending -> in_progress -> completed/failed)
    - Use color codes (green for completed, red for failed, yellow for in_progress) via ANSI escape codes
    - Run until interrupted (Ctrl+C), handle KeyboardInterrupt gracefully

    `show_status(state_file_path)`:
    - One-shot display of current state
    - Show all tasks with: task_id, status, skill_hint, created_at, last activity entry
    - Show active container count (from tasks with status in_progress/starting/testing)
    - Format as a simple table for terminal readability

    `show_task_detail(state_file_path, task_id)`:
    - Show full activity log for a specific task
    - Include all timestamped entries
    - Show task metadata (skill_hint, staging branch, runtime)

    **CLI entrypoint** (at bottom of file):
    ```python
    if __name__ == '__main__':
        import argparse
        parser = argparse.ArgumentParser(description='OpenClaw L3 Monitor')
        subparsers = parser.add_subparsers(dest='command')

        # openclaw-monitor tail
        tail_parser = subparsers.add_parser('tail', help='Stream L3 activity in real-time')
        tail_parser.add_argument('--interval', type=float, default=1.0)

        # openclaw-monitor status
        status_parser = subparsers.add_parser('status', help='Show current L3 status')

        # openclaw-monitor task <task_id>
        task_parser = subparsers.add_parser('task', help='Show task detail')
        task_parser.add_argument('task_id', type=str)

        args = parser.parse_args()
        # ... dispatch to functions
    ```

    The monitor reads state.json with shared locks (LOCK_SH) via the state engine, so it never blocks L3 writers. It is a read-only observer.

    Keep the implementation simple — plain Python, no external dependencies beyond the orchestration package. This is a Phase 3 interim tool; the full dashboard comes in Phase 4.
