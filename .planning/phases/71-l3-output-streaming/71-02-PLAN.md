2. In `route.ts` (`/api/events`), add a heartbeat timer inside the ReadableStream `start()` callback:
   - After the `client.on('connect', ...)` handler, set up a 30-second interval that sends SSE comment pings:
     ```typescript
     const heartbeatInterval = setInterval(() => {
       try {
         controller.enqueue(encoder.encode(': ping\n\n'));
       } catch (e) {
         // Controller closed — clean up
         clearInterval(heartbeatInterval);
       }
     }, 30_000);
     ```
   - Clear the interval in the `client.on('close', ...)` handler and in the `abort` event handler:
     ```typescript
     client.on('close', () => {
       clearInterval(heartbeatInterval);
       // ... existing close logic
     });

     request.signal.addEventListener('abort', () => {
       clearInterval(heartbeatInterval);
       // ... existing abort logic
     });
     ```
   - Also handle the Python-side heartbeat messages: in the `client.on('data', ...)` handler, check if the parsed JSON has `type: "heartbeat"` and if so, send it as an SSE comment (`: ping`) rather than as an event, to avoid polluting the event stream:
     ```typescript
     client.on('data', (data) => {
       const lines = data.toString().split('\n').filter(Boolean);
       for (const line of lines) {
         try {
           const parsed = JSON.parse(line);
           if (parsed.type === 'heartbeat') {
             controller.enqueue(encoder.encode(': ping\n\n'));
             continue;
           }
         } catch (e) {
           // Not JSON — forward as-is
         }
         controller.enqueue(encoder.encode(`event: message\ndata: ${line}\n\n`));
       }
     });
     ```

3. Add the `Last-Event-ID` header support for reconnection: When the SSE client reconnects, it sends `Last-Event-ID`. Add an incrementing event ID to each SSE message:
   - Add a `let eventId = 0;` counter in the `start()` scope.
   - Change the event emission to include the id field:
     ```typescript

     ```
   - Note: Full buffered history replay (last 100 events per task) is a server-side concern. For this phase, add a simple in-memory ring buffer (100 events) in the route handler scope. On reconnect (when `Last-Event-ID` header is present in request), replay buffered events after the given ID before switching to live streaming. Store events as `{id: number, data: string}` tuples.
