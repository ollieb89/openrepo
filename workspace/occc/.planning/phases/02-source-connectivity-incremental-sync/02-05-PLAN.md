  <name>Task 1: Implement Background Sync Logic</name>
  <files>
    - src/lib/sync/scheduler.ts

  <action>
    Create a server-side scheduler utility in `src/lib/sync/scheduler.ts`.
    
    1. Define `BACKGROUND_SYNC_INTERVAL_MS` (default 1 hour / 3600000ms).
    2. Implement `runBackgroundSync()`:
       - Fetch all connectors using `listConnectorStates()`.
       - Filter for connectors that are `enabled === true`.
       - Filter out connectors with status: `syncing`, `rate_limited`, or `auth_expired`.
       - Check `lastSyncedAt`: If `(now - lastSyncedAt) < BACKGROUND_SYNC_INTERVAL_MS`, skip.
       - For eligible connectors, call `runIncrementalSync({ connectorId })` from `@/lib/sync/engine`.
       - Implement a simple lock (e.g., a module-level variable) to ensure only one `runBackgroundSync` execution happens at a time.
    3. Add basic logging for scheduler start/skip/finish.
  </action>
  <verify>
    Create a temporary test script or use a console log to verify that `runBackgroundSync` identifies due connectors and triggers their sync while skipping healthy/recent ones.
  </verify>
  <done>
    `runBackgroundSync` correctly identifies and triggers sync for eligible connectors and respects the 1-hour interval and health-state guards.
  </done>
</task>

<task type="auto">
  <name>Task 2: Create Background Sync API Route</name>
  <files>
    - src/app/api/connectors/sync/background/route.ts
  </files>
  <action>
    Create a POST endpoint in `src/app/api/connectors/sync/background/route.ts`.
    
    1. The handler should call `runBackgroundSync()`.
    2. It should return a JSON response summarizing the action:
       - `triggered`: list of connector IDs that started syncing.
       - `skipped`: count of connectors skipped due to health or interval.
       - `status`: "success" or "busy" (if lock is active).
    3. Ensure it doesn't wait for all syncs to complete if they are long-running (fire and forget or run in background if runtime supports it, but since we are in Next.js, `runBackgroundSync` should probably be awaited if we want to ensure it finishes within the request, or use `waitUntil` if on Vercel. For local/desktop, a standard await is usually fine for a few connectors).
  </action>

  <done>
