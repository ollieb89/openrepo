# Plan 56-02: Docker Pause Loop & Orchestrator Notification

## 1. Objectives
- Prevent the L3 container from exiting immediately upon escalation.
- Implement a loop that waits for manual/L2 intervention.
- Catch the escalation event in the orchestrator to trigger an alert.

## 2. Context
Currently, the runner calls `sys.exit(1)` when things fail completely. For an escalated task, we want the container to stay alive (paused) so that the human or Orchestrator can debug it or send a resume signal without losing state. We also need to notify the user.

## 3. Requirements
- When `ESCALATING` state is reached, the runner must enter an `asyncio.sleep()` loop.
- The runner must periodically check for a `resume` signal (e.g., checking if `JarvisState` status is back to "executing", or a specific `unpause` file appears).
- The Orchestrator (`hooks.py`) must listen for `AutonomyEscalationTriggered`.
- The Orchestrator must log a high-priority simulated direct ping containing the task ID and confidence score.

## 4. Tasks

1. **Implement Pause Loop in `runner.py`**:
   - Create `async def _escalation_pause_loop(self)`.
   - The loop checks the task state via `JarvisState(self.state_file)`.
   - If state changes from "escalating" to "executing" or "resumed", break the loop and continue (or trigger a replan).
   - For v1 MVP, simply allow breaking the loop to retry the current step or gracefully exit if told to terminate.
   - Replace `sys.exit(1)` in the escalation block with `await self._escalation_pause_loop()`.

2. **Add Host-Side Event Listener**:
   - In `packages/orchestration/src/openclaw/autonomy/hooks.py`, subscribe to `EVENT_ESCALATION_TRIGGERED`.
   - Add handler `_on_escalation(envelope)`.
   - Log the direct ping alert: `[TELEGRAM_PING] Task {task_id} Escalated! Reason: {reason}`.

3. **Update Unit Tests**:
   - Modify `test_runner.py` to mock `_escalation_pause_loop` so tests don't hang, or use a side_effect that sets the resume flag after 1 iteration.
   - Assert the pause loop is called when threshold is breached.

4. **Integration Testing Notes**:
   - Ensure `spawn.py` timeout values or container constraints don't prematurely kill the container while it's in the pause loop (though standard L3 config sets restart policy to `no`, the script just sleeping shouldn't trigger an automatic kill unless L2 explicitly kills it).

## 5. Success Criteria
- The runner successfully enters a polling wait state when escalated.
- The Orchestrator logs the simulated Telegram notification.
- An external process can break the runner out of the loop by modifying the state file.
